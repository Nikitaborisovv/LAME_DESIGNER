# ARCHITECTURE.md — принципы, модель данных и дорожная карта

Единый источник правды по **целевой** архитектуре и плану перехода к ней. Конвенции
текущего кода (как писать в этом репозитории сейчас) — в [CLAUDE.md](CLAUDE.md); этот файл —
про принципы, диагноз производительности и куда движемся.

> Проект вырос из «flat/cloud + vision-воркер» в систему **слоёв-продюсеров + top-level
> эффектов + драйверов** с нод-редактором (Rete) как проекцией поверх той же модели. Это уже
> фактически граф в духе TouchDesigner/Houdini. Цель — довести его до realtime-класса
> (60 fps рендера при полном стеке эффектов), оставаясь в вебе/десктопе.

## 0. Два неизменных принципа

Эти два принципа не меняются ни на одной фазе. Всё остальное — частный случай.

### 0.1. Разделение частот

Рендер (Three.js, шейдеры) идёт на 60 fps. Тяжёлые вычисления (нейронки, инференс, CV) НЕ
считаются каждый кадр — они работают на своей, более низкой частоте (воркеры / детекторы вне
`useFrame`), а рендер переиспользует последний результат через `ref`. **Никогда не вызывать
инференс синхронно в `useFrame`.** Точки управления частотой: `depthEveryNthFrame`,
`peopleEveryNthFrame`, busy-флаги воркеров, троттлинг по timestamp, круговой планировщик
детекторов (`core/scheduler.ts`).

В целевой модели обобщается: **у каждого узла графа свой клок**. Продюсеры наполняют данные
на низкой частоте, операторы трансформируют, синки рисуют на 60 fps по последнему результату.

### 0.2. GPU-resident: не гонять per-element через CPU/DOM каждый кадр

Главный урок TouchDesigner (см. §8, оба TD-справочника) и **причина наших текущих тормозов**
(§2). Правила:

- **Держи данные на GPU как можно дольше.** Самое дорогое — переходы GPU↔CPU (чтение пикселей,
  `TOP→CHOP`-аналоги). В нашем стеке это: readback карт/масок на главном потоке и **рендер
  оверлеев через SVG/DOM**.
- **«Поэлементная раздача» — самый медленный метод.** В TD это export канала на каждую ноду;
  у нас — пересборка `svg.innerHTML` строкой по каждой точке каждый кадр (constellation,
  faceMesh 468×лиц, линии). Это чистый CPU + парсинг DOM. **Замена — off-DOM иммедиэйт-рендер,
  гибридно по числу элементов:** `Canvas2D` для оверлеев с малым числом элементов и богатой
  стилизацией (руки/лица/HUD — точная паритетность дёшево); **WebGL-геометрия** (инстансы +
  line-segments, позиции в буферах) для масштабных (constellation/частицы, O(N²)) и 3D-унификации.
- **Не читай layout в кадре.** `clientWidth`/`getBoundingClientRect` в rAF-цикле форсируют
  синхронный reflow; рядом с любым DOM-оверлеем это **layout-thrash** — стоимость reflow
  «прилипает» к тому, кто читает размер. Размер кешируй через `ResizeObserver`. Это критично:
  замер `hands.draw` 15мс→**0.16мс** (≈95×) получился только вместе со снятием layout-чтения —
  один Canvas2D без этого в смешанной сцене НЕ помогает (reflow просто перескакивает на соседа).
- **Постоянная топология + меняющиеся позиции = дёшево.** Кто с кем соединён (рёбра меша,
  скелет руки, цепь) не пересобираем каждый кадр — гоним только координаты в буфер.
- **Состояние между кадрами на GPU = feedback/ping-pong**, а не CPU-переменные (для будущих
  частиц — §3.4).
- Это «вебовый POP-принцип»: realtime-потолок задаётся тем, насколько редко данные пересекают
  границу GPU↔CPU.

## 1. Модель данных: три типа + параметры

Мысль TD: «всё — это числа; семейства операторов — лишь разные представления». Заводим три
типа данных (+ параметры) и **конвертеры** между ними.

| Тип | Аналог TD | В проекте |
|---|---|---|
| **Map2D** | TOP (пиксели, GPU) | видео, глубина, рампы, маски, отрендеренные точки |
| **PointSet** | SOP/POP (точки + атрибуты) | движение, лица, боксы, руки, облако глубины |
| **Field** | CHOP (сигнал во времени) | энергия движения, состояние жеста → управление параметрами |
| **Params** | — | `SceneConfig` (единый источник правды по значениям) |

### 1.1. PointSet — модель атрибутов (Houdini-стиль, SoA)

Structure-of-arrays из типизированных массивов: дёшево и transferable между воркерами; и —
важно для §0.2 — позиции уже лежат буфером, готовым уехать прямо в WebGL-атрибут.

```ts
interface PointSet {
  count: number;
  attrs: {
    P: Float32Array;        // позиции xy (или xyz с глубиной), itemSize 2/3
    id?: Float32Array;      // стабильный ID (трекинг боксов между кадрами, IoU)
    group?: Uint16Array;    // источник/инстанс (лицо #2, рука L, бокс #3)
    conf?: Float32Array; vel?: Float32Array; Cd?: Float32Array;
    N?: Float32Array; size?: Float32Array; age?: Float32Array;
  };
  edges?: Uint32Array;      // топология: сетка лица/скелет руки дают её готовой; "nearest" считается
  space: "uv" | "world";
}
```

Стандартные имена (`P`, `Cd`, `N`, `id`, `vel`…) переиспользуются всеми операторами — эффект
не знает, какой продюсер их породил. Сейчас в коде упрощённый предшественник — `PointField`
(`core/types.ts`): группы `{points, links?, chain?, values?, color?}`. Обобщение
`PointField → PointSet` — на дорожной карте (§3).

### 1.2. DataBus — адресуемость без ручной разводки

Каждый активный продюсер публикует свои выходы под стабильными хендлами с объявленным типом;
потребитель выбирает хендл (в графе — провод). Резолвер — `fieldForKind` в `core/pointSources.ts`.

```
motion → motion.points · motion.boxes · motion.heat(Map2D)
faces  → faces.mesh(+edges) · faces.box
hands  → hands.points(+skeleton)
people → people.boxes · people.mask(Map2D)
depth  → depth.map(Map2D)        video → video.color(Map2D)
```

## 2. Где мы сейчас + измеренный диагноз производительности

### 2.1. Текущая модель (что в коде)

`LayeredConfig` = глобалы + `layers[]` (продюсеры/эффект-шейдеры) + `effects[]` (top-level
ноды-потребители точек, пока «линии», с хендлом `source`) + драйверы (биндинг сигнала на
параметр). `resolveConfig` сворачивает слои в плоский `SceneConfig`, который читает весь
рендер. Нод-редактор (`ui/GraphView.tsx`, Rete v2) — **проекция поверх той же модели**, не
отдельное состояние. Подробности конвенций — в [CLAUDE.md](CLAUDE.md).

### 2.2. ЗАМЕР (2026-06-05, профайлер `__perf`)

Полный стек включён (sobel+thermal+pixelArt+motion+constellation2D+faceBoxes+faceMesh 468×+
hands+peopleMask(YOLO)+selfie+peopleBox+hud + 5 эффектов-линий), видео `videos/IMG_3532.MOV`.
Результат: **`render.fps` ≈ 7**. Разбивка главного потока (ema, мс/кадр):

| Категория | Стадии | Сумма | Что это |
|---|---|---|---|
| 🔴 **SVG-оверлеи** | `overlay.draw` 16.6 + `hands.draw` 15.3 + `modifiers.draw` 4.8 | **≈37 мс** | `innerHTML` строкой каждый кадр (поэлементная раздача) |
| 🔴 **MediaPipe sync на main** | `faces.detect` 22 + `hands.detect` 17 + `seg.detect` 10 | **≈15–22 мс/кадр** | синхронный инференс блокирует кадр; `seg` идёт мимо планировщика |
| 🟡 **Readback** | `vision.readback` 8.9 + `people.readback` 5 | **≈14 мс** | каждый хук перерисовывает видео в свой канвас и читает пиксели |
| 🟢 **сам WebGL-рендер** | `render.js` | **0.07 мс** | **GPU простаивает** |

`vision.roundtrip` 85мс / `people.roundtrip` 68мс — **симптомы** забитого главного потока
(результат воркера не подхватывается вовремя), а не причина: в самих воркерах компьют дёшев
(`vision.worker` 3.6мс; `people.run` 38мс в своём потоке).

### 2.3. Вердикт

**Мы на 100% CPU/main-thread bound. GPU не загружен (`render.js` 0.07мс).** Следствия,
определяющие приоритеты:

- Текущий лаг — **архитектурный CPU**: DOM-оверлеи + синхронный инференс. Не потолок GPU.
- **Переезд в TouchDesigner текущий лаг НЕ решает** — проблема не там. (Решение по итогам
  обсуждения: десктоп-доставка ОК, но переписывать всё в TD ради этого лага — лечить насморк
  ампутацией; цель «миллионы частиц» к нынешнему лагу отношения не имеет и упрётся только
  после чистки CPU.)
- Всё чинится в текущем стеке применением §0.2.

## 3. Дорожная карта (по убыванию рычага)

### P1 — оверлеи SVG → off-DOM (Canvas2D / WebGL) ⬅ **СЕЙЧАС** (убирает ≈37 мс/кадр)

Перенести рендер точек/линий из `innerHTML`-SVG в иммедиэйт-рендер. Решение (см. §0.2):
**гибрид** — `Canvas2D` для оверлеев с малым числом элементов и стилизацией (руки/лица/HUD),
WebGL-геометрия для масштабных (constellation/частицы, O(N²)) и 3D. Координаты — тем же
«contain»-фитом, что шейдер. **Размер — через `ResizeObserver`, не читать `clientWidth` в кадре.**

- ✅ **2D-оверлеи перенесены на Canvas2D** (`HandsOverlay`, `Overlay`, `ModifierOverlay` +
  `renderLines.drawLines`, `serpCurve`). Все — с `ResizeObserver`, без чтения layout в кадре.
  Замер (та же тяжёлая сцена, что §2.2):

  | стадия | SVG | Canvas2D |
  |---|---|---|
  | `overlay.draw` | 15.6мс | **2.9мс** |
  | `modifiers.draw` | 4–18мс | **0.3мс** |
  | `hands.draw` | 15.3мс | **0.01мс** |
  | **`render.fps`** | **7** | **19** (≈2.7×) |

  Урок (важно для будущих оверлеев): выигрыш появился только вместе со снятием per-frame чтения
  layout (`clientWidth`) — иначе reflow-тэкс соседних SVG «прилипает» к канвасу (при половинной
  миграции `modifiers.draw` подскакивал 4→18мс). Constellation ≤200 и faceMesh 468 на Canvas2D
  идут за единицы мс — WebGL понадобится только при тысячах+ точек.
- ⬜ Осталось: `ui/Constellation3D.tsx` (3D-оверлей, `innerHTML` каждый кадр; только в cloud-режиме).
- **Узкое место сместилось** (новый верх, → P2/P3): `vision.readback` ~23мс (видео→канвас→
  `getImageData` для motion-воркера на главном потоке) + синхронный MediaPipe (`faces.detect` 12мс,
  `seg.detect` 6.6мс, `hands.detect` 6мс). Оверлеи больше НЕ узкое место — диагноз §2 подтверждён.

### P2 — MediaPipe (после P1 — главный остаток, ~10–15 мс/детектор)

> ⚠️ **Ключевой факт (замер):** после P1 система **GPU-bound на синхронных MediaPipe-вызовах**
> (`faces.detect` 12–15мс, `hands.detect` 7–10мс, `seg.detect` 9–12мс — все GPU-delegate на main).
> Любая ДОБАВЛЕННАЯ GPU-работа их замедляет (см. P3: `createImageBitmap`-ресайз поднял
> `faces.detect` 12→28мс через GPU-контеншен). Поэтому разгружать CPU за счёт GPU — НЕЛЬЗЯ.

- ✅ **`seg` заведена в круговой планировщик** (`App.tsx`): раньше шла мимо (каждый 2-й кадр
  синхронно) → могло быть до 2 MediaPipe-вызовов/кадр; теперь строго ≤1 (faces/hands/seg
  делят ротацию, budget 1). `seg.submit` уже возвращает boolean — контракт `SchedTask`.
- ⬜ Снизить GPU-стоимость детекторов (пользовательский tradeoff качество↔fps): меньше входное
  разрешение, меньше `maxFaces`/`maxHands`, реже частота. Это единственный рычаг без воркеров.
- ⬜ Перенос в воркеры заблокирован в **Vite-dev** (MediaPipe `FilesetResolver` динамически
  `import()`-ит wasm-лоадер, Vite отдаёт его как `…?import` → issue #5292; classic-воркер не
  может top-level `import` пакета). Пути: (а) self-host wasm-лоадера мимо vite-резолвера;
  (б) classic-воркер-бандл (esbuild IIFE) только для MediaPipe; (в) **проверить prod-сборку**.

### P3 — readback кадра ✅ один грабер кадра

`drawImage(<video>)` в CPU-канвас тащит ВЕСЬ кадр высокого разрешения (1080p/4K) в RAM (~22мс,
downscale-фильтрация ни при чём — проверено `imageSmoothingEnabled=false`). Раньше эту выемку
делал КАЖДЫЙ CPU-детектор (vision 23мс + people 4.8мс + seg внутри `seg.detect`) — 3×/кадр.
- ✅ **`ml/useFrameGrab.ts`:** один `drawImage(<video> → общий CPU-канвас 640)` на кадр
  (`frame.grab` ~22мс, 1×); vision/seg/people рисуют уже из него (CPU→CPU, мелко). Результат:
  `vision.readback` **23→0.4мс**, `people.readback` **4.8→1.2мс**. faces/hands НЕ через грабер
  (MediaPipe грузит видео на GPU сам). seg — в планировщике, рисует из грабера.
  - ⚠️ **Грабер ОБЯЗАН быть CPU-backed (`willReadFrequently`).** Если GPU-backed — каждый
    CPU-детектор форсирует свой GPU→CPU ридбэк 640-кадра, всё клинит до 1fps (проверено).
  - Структурный выигрыш: дорогая выемка теперь O(1), а не ×(число CPU-детекторов); **без**
    GPU-контеншена (в отличие от `createImageBitmap`).
- ❌ **`createImageBitmap` с GPU-resize — отклонён замером:** ридбэк 23→2мс, НО ресайз на GPU
  контендит с MediaPipe (faces 12→28мс) → чистый регресс (система GPU-bound, P2).
- ⬜ Остаток: сама `frame.grab` ~22мс **неустранима** без (а) троттлинга (motion реже — quality↔fps)
  или (б) GPU-пути (контеншен). `render.fps` теперь упирается в `frame.grab` + MediaPipe.

### P4 — нод-редактор: остаток (UX, после перфа)

- ✅ **Провод vs ссылка (принцип 2, TD wire/reference).** В графе два вида связей:
  ПОТОК (сплошной провод, points/map) и ССЫЛКА (пунктир, field/жёлтый) — драйвер-сигнал руки
  переопределяет ПАРАМЕТР эффекта (TD export-to-param). Нода «Сигналы рук» (Field) с выходами
  openness/pinch/roll/handsDist; на эффект-ноде — ref-входы параметров (`bindable`). Создание
  ссылки пишет `effect.bindings[param]` (lo/hi из спеки, smooth 0.6), снятие — чистит
  (`connectionremoved`, гард `ref` + «нет замены»). Тип-гард уже блокирует points→param.
  **direction A (2026-06-10):** ref-входы появились и на shader-FX-нодах (`LAYER_DEFS[kind].bindable`,
  пишут `Layer.bindings[param]`) — детали в §3.6.9, блок «direction A».
  Двусторонняя паритетность с `BindingRow` инспектора (общий `setEffectBinding`). Легенда в графе.
  Сигналы сгруппированы по нодам-источникам (`DRIVER_GROUPS`): «Сигналы рук», «Движение · энергия»,
  «Лица · центроид» (faceX/faceY), «Люди · число» (peopleCount). Драйверы НЕ только руки — это
  Reduce'ы любых продюсеров (§4.3): энергия движения, центроид лиц, count людей; работают без рук.
  Источники единые с точками (`computeDrivers(DriverSources)` ← тот же DataBus-вход, что `sources`).
- ✅ `connectionremoved` для ПОТОКА точек → **дизейбл эффекта** (нет входа — нечего потреблять;
  `source` non-nullable, поэтому помечаем `enabled=false`, а не зануляем). Обратное подведение
  потока (`connectioncreated` points) снова включает. Гард «нет замены» (как у ссылок) +
  `nodeToKind.has(source)`, чтобы `enforceSingleInput`/мисматч `field→points` не дизейблили.
- ✅ Действия на ноде-эффекте (кастом-контрол Rete `NodeActions`): ⏻ дизейбл + ✕ удаление —
  раньше удаление было только незаметной клавишей Delete. `onPointerDown→stopPropagation`, чтобы
  клик кнопки не уводил в драг/пик ноды. (Тумблер держит локальный стейт; дизейбл по снятию
  `points`-потока обновит его только при пересборке графа — мелкая рассинхронизация, не критично.)
- 🔶 Превью выхода на ноде (вьюер): ✅ живой CHOP-вьюер на нодах-сигналах (`SignalMeters` —
  полоска+число на канал, обновление на rAF БЕЗ React-ререндера, пишем в DOM по ref, §0.2; читает
  общий `getDrivers()` из живых рефов продюсеров). ⬜ Осталось: миниатюра выхода эффект-ноды (Map2D)
  + пресеты на ноду/граф.
- **Кит утилитарных нод** (§4): Math/Remap, Group/Filter, Merge, Range — базовый набор.
- Ревизия входов/выходов КАЖДОЙ ноды по смыслу (не только points/map/field).
- ✅ Авто-активация зависимостей (TD-dependency: потребитель заставляет продюсера cook).
  `resolveConfig` после свёртки слоёв форс-включает продюсеров, на которых ссылается ВКЛЮЧЁННЫЙ
  эффект: `source` (через `PRODUCER_ENABLE`) и продюсеров его ссылок-драйверов (`bindings` →
  `driverProducer`). Эффект с `source=faceMesh` сам поднимает детектор лиц; ссылка на `peopleCount`
  поднимает YOLO. Выключенный эффект не форсит. (Каверза: детекция и оверлей продюсера гейтятся
  одним флагом → попутно включается и его оверлей. Разделение «cook vs display» — на будущее.)

### P5 — масштаб: WebGPU-compute для частиц/точек (миллионы + физика)

> ⤳ **Развёрнуто в §3.6.6 (трек T5)** — полная архитектура частиц на миллионы (эмиттеры/силы/скаттер/
> коллизии-об-глубину/свет/рендер, WebGL2→WebGPU). P5 = бэкенд-этап этого трека.

Когда CPU вычищен и упрёмся в число элементов — поднять **WebGPU-compute (TSL в Three)**:
web-аналог `Particle POP`. Состояние частиц живёт в VRAM (storage-буферы SoA), физика —
compute-шейдер, рендер — Points/инстансингом, **без возврата на CPU в кадре** (§0.2).
**Ревизия 2026-06-10: TSL-first** — TSL транспилируется и в WGSL, и в GLSL (WebGL2-фолбэк через
transform feedback автоматически), свой RTT-ping-pong-солвер как основа больше не нужен.
Частицы — изолированный модуль на `three/webgpu` (ядро на `ShaderMaterial` не мигрирует).
Детали и цифры — §3.6.6. WebGPU у нас уже в проде (глубина через transformers.js) — бэкенд знаком.

- **Модель данных:** продюсеры точек (`pointSources.ts`) + DataBus-резолвер `fieldForKind`;

- **Нод-редактор (Rete v2):** граф над живой сценой (скрим, панели не прячутся); типизированные

- **Провод vs ссылка (принцип 2):** ПОТОКИ (сплошной провод) и ССЫЛКИ (пунктир-field) разведены —
  нода «Сигналы рук» + ref-входы параметров на эффекте; драйвер→параметр пишет/чистит `bindings`

- **Эффекты как top-level ноды (keystone):** `Layer.modifiers[]` убран → `LayeredConfig.effects:
  EffectNode[]` (`{id, kind, enabled, source, params, bindings?}`, `source` без «self»); миграция
  backward-compat `drainModifiers`+`sanitizeEffects` в `presets.merge`; `EffectInspector` при
  выборе ноды-эффекта. Подзадача 2-A: встроенные line-параметры (`motionLink*`) и рендер
  цепь-линий выпилены из продюсера движения — теперь это отдельная нода «Линии» (`source=motion`).

## 3.5. РАЗВОРОТ: граф как источник правды (слои → ноды) ✅ **СДЕЛАНО (A→C)** — далее §3.6

> Решение (по итогам сессии): **модель слоёв как UX не работает — убираем**. Нод-редактор из
> «проекции» становится **источником правды**. Рендер ведёт граф, а не резолвнутый `SceneConfig`.
> Подтверждено: миграция фазами (**структура сначала**, всё время рабочее) + **чистый граф**
> (детекция отдаёт ТОЛЬКО данные; чтобы увидеть — тянешь в ноду-рисовалку → Экран).

### 3.5.1. Таксономия нод (одна нода на смысл, мульти-выходы — НЕ плодить 10 нод на эффект)

| Нода (тип) | Входы | Выходы | Настройки (правый инспектор) |
|---|---|---|---|
| **Видео** (источник, корень) | — | `видео`(map) · `глубина`(map) · `облако`(points) | источник файл/телефон/Kinect; глубина: разрешение/сглаживание/кеш |
| **Руки** (детект) | `видео`(map) | `точки`(points) · `раскрытие`/`щипок`/`поворот`/`дистанция`(field) | maxHands, зеркало, сглаживание |
| **Лица** (детект) | `видео`(map) | `сетка`(points) · `боксы`(points) · `центр X`/`центр Y`(field) | maxFaces, плотность, сглаживание |
| **Люди** (детект) | `видео`(map) | `маска`(map) · `боксы`(points) · `число`(field) | модель, conf, порог |
| **Движение** (детект) | `видео`(map) | `боксы`(points) · `тепло`(map) · `энергия`(field) | чувствительность, затухание |
| **Тепловизор / Sobel / Пиксель-арт** (2D-FX) | `map` (Пиксель-арт +`глубина`) | `map` | свои крутилки |
| **Линии / Splat** (рисовалки) | `points` | `map` | топология/форма/цвет |
| **Math / Reduce / Scatter / Sample** (конвертеры) | field / points / map | field / points | gain·curve / count / атрибут |
| **Экран** (выход, сток) | `map` | — | — |

Ключевое: **сигналы-драйверы — это field-выходы детект-нод** («Сигналы рук» сливаются в «Руки»;
центроид — в «Лица»; энергия — в «Движение»; число — в «Люди»). Детекция = ОДНА нода с
несколькими типизированными выходами, а не куст. 2D-эффекты — `map→map`, чейнятся проводами.

### 3.5.2. Сокеты и правила связей
- Типы кабелей (один тип на провод, уже есть): `map`(розовый) · `points`(голубой) · `field`(жёлтый).
- Поток = сплошной провод; ссылка-драйвер (field→параметр) = пунктир (уже есть).
- Матрица: детект-ноды требуют `видео`(map); 2D-FX `map→map`; рисовалки `points→map`;
  конвертеры по своим типам; всё сходится в `Экран`(map). Разнотипное не соединяется (есть).

### 3.5.3. Схема конфига-графа (целевая)
`GraphDoc = { nodes: Node[]; edges: Edge[]; globals }`, где `Node = {id, kind, params, pos}` и
`Edge = {from:{node,out}, to:{node,in}}`. Связи **явные** (вместо неявного `effect.source` и
`layer.enabled`). Компилятор `compile(GraphDoc)` каждый кадр даёт: активные детекторы (по наличию
рёбер от «Видео»), что какая рисовалка потребляет, цепочку 2D-FX, ссылки-биндинги. Старый
`LayeredConfig` мигрируется в `GraphDoc` (back-compat загрузки пресетов).

### 3.5.4. UI
- **Tab** → палитра добавления нод, делённая по категориям (Источники/Детекция/2D-FX/Точки/
  Конвертеры/Поле/Выход). **Правая панель = инспектор выбранной ноды** (обобщение `EffectInspector`).
  **Панель слоёв удаляется.**

### 3.6.0. Суть парадигмы (почему так — 5 уроков, к ним всё сводить)
1. **Недеструктивность.** Каждая нода берёт данные на вход и возвращает НОВЫЕ; источник не мутируется.
   Бесконечная переразводка, живые параметры (= наш принцип «без скрытого состояния»).
2. **Типизированные провода + атрибуты текут с данными.** Провод несёт тип; геометрия несёт ИМЕНОВАННЫЕ
   атрибуты (P/Cd/N/pscale/orient/id/v/age/life), которые «едут» вниз по графу и читаются downstream —
   без глобальных сайд-каналов.
3. **Поля (fields) рулят всем.** Сокет несёт либо число, либо *рецепт* «посчитай это per-element».
   ОДНО выражение масштабируется на миллион элементов без явных циклов; ОДНО поле может рулить и
   позицией, и цветом, и выделением, и плотностью — «написал раз, переиспользуй как сигнал везде».
4. **Selection — это ВХОД, а не нода.** Почти у каждого потребителя есть опц. вход `selection`(Field/маска):
   булево/0..1 поле выделяет подмножество. Та же маска РЕЖЕТ (delete/split), ограничивает ИНСТАНСИНГ,
   гейтит ЦВЕТ, весит MIX. Один mask → много поведений (главный множитель силы — урок Blender).
5. **Состояние = фидбэк по буферу.** Частицы/трейлы — это `state_{t+1}=integrate(state_t,forces,dt)`,
   где state живёт в FBO ping-pong (движок уже есть в `FlatView`), а не отдельный «движок частиц».

И сквозной урок TD: **границы типов (Map2D↔PointSet↔Field, GPU↔CPU) — это ЯВНЫЕ ноды-конвертеры**, а
не скрытые конверсии. Дорогие переходы (CLAUDE.md §2) видны в графе.

### 3.6.4. Принципы дизайна нод (УТОЧНЕНО пользователем — главное)
1. **Ноды — по СМЫСЛУ. Enum — ТОЛЬКО для вариаций ОДНОГО действия.** ✅ enum уместен: `Math`(add/mul/min/…),
   `Composite`(blend-режим), `Noise`(тип). ❌ НЕ сжимать разные по смыслу в мега-ноду: **Ramp ≠ Composite ≠ Mix**
   (три ноды); **примитивы — отдельные по форме** (Box/Grid/Sphere/Polygon, а не один «Primitive» с shape-enum;
   ВНУТРИ ноды форма параметрическая — `Polygon` морфит по `sides`); **силы — отдельные** (Gravity/Curl/Attract,
   не один «Force»); **спрайты-2D ≠ примитивы-3D**. Правило: enum — вариация действия; отдельная нода — отдельное
   действие/форма (как в Blender/Houdini: Cube/Grid/Circle отдельные, Math — один с enum).
2. **Selection — это ВХОД, а не нода** (везде опц. `Field`-маска: режет/инстансит/красит/миксует).
3. **Границы типов (Map2D↔PointSet↔Field) — явные ноды-конвертеры.**
4. **Удобство = организованная Tab-палитра, а НЕ «меньше нод».** Много осмысленных нод, но мгновенно находимых:
   **категории-по-контексту + поиск-по-набору (печатаешь «ramp» → фильтр) + цвет по типу выхода** (Map2D-розовый/
   PointSet-голубой/Field-жёлтый). Опц. вкладки-типы, внутри — категории. (Houdini Tab-меню = образец.)

### 3.6.5. Примитивы по алгоритму (не модели)
Один дескриптор `{shape, sides, radius[xy(z)], thickness/onion, height, rows/cols}` питает И 2D-SDF-спрайт,
И 3D-меш-с-точками. `sides`: 3→треугольник, 4→квадрат, ∞→круг; `thickness`/onion → контур/«квадрат с дыркой»
(`d=abs(d)-w`, §5.1); растяг по оси → прямоугольник/эллипс. 3D-примитив несёт СВОИ точки → их можно
copy-to-points дальше. Цепочка: спрайты на точки куба → кубы на точки облака (но только на группу, срезанную рампой).

### 3.6.6. Частицы на МИЛЛИОНЫ (GPU) — отдельный приоритетный трек ⭐ (T5)

> ⚠ **РЕВИЗОВАН → §3.6.12** (2026-06-10, после спайка и установки пользователя «система, не фичи»):
> актуальный спек нод (Emitter points+map, Force одной нодой с секциями, реалтайм-инварианты,
> фазы A–D) — там. Этот раздел — исходный дизайн, верен по физике/цифрам; при расхождении верь §3.6.12.

> **Ревизия 2026-06-10 (разведка §3.6.11):** прежний план «сначала свой WebGL2-RTT-ping-pong, потом WebGPU»
> заменён на **TSL-first**. Факт: TSL (Three Shading Language) официально транспилируется И в WGSL (WebGPU),
> И в GLSL (WebGL2-фолбэк); `WebGPURenderer` сам падает в WebGL2-бэкенд, где `compute()` реализован через
> transform feedback. То есть «один код нод → два бэкенда» уже существует в three.js — строить свой
> ping-pong-солвер как ОСНОВУ значит делать то, что потом выбросим. Ограничение: `WebGPURenderer` НЕ
> поддерживает `ShaderMaterial` → ядро рендера (`flatShader`/FBO-чейн) НЕ мигрируем; **частицы — изолированный
> модуль на `three/webgpu` + TSL** (отдельный канвас/слой поверх, фичефлаг), композит — поверх flat-кадра.

- **State — SoA storage-буферы по атрибуту** (TSL `instancedArray`: pos vec3 · vel vec3 · life/seed vec2;
  1М × 32 байта = 32 МБ — ок). Это модель TD POPs (SSBO на атрибут), она же — наш §1.1 PointSet. На
  WebGL2-фолбэке тот же TSL-код ложится на transform-feedback-буферы автоматически.
- **Дизайн под общий знаменатель фолбэка:** базовый апдейт (позиция/скорость/age, силы из текстур) — только
  element-wise операции, БЕЗ атомиков/shared memory → один TSL-`Fn` работает на обоих бэкендах. Соседские
  фичи (flocking/SPH/частица-частица — uniform grid, сортировки) — capability-гейт, **WebGPU-only**.
- **Solver** (нода): ЕДИНСТВЕННЫЙ владелец рождения/смерти (урок TD Particle POP). Semi-implicit Euler
  (`v+=a·dt; P+=v·dt`), clamped-dt (`min(Δ,1/30)`), **age-recycle**: фикс. популяция, мёртвая частица
  респавнится сэмплом эмиттер-карты — целиком на GPU, без компакции/free-list. Инвариант петли: ноды-силы
  число частиц НЕ меняют.
- **Силы** (отдельные ноды; схема Houdini POP): каждая сила **аккумулирует атрибут `force`**, интегратор
  один и стандартный → порядок force-нод не важен. Набор: `Gravity`·`Drag`(`-k·v`)·`Wind`·
  **`Curl/Turbulence`** (дивергентно-свободный curl-noise по Bridson; дешёвый вариант — bitangent noise,
  ~2× быстрее)·`Vortex`·`Attract/Repel`(1/r²; аттракторы = точки рук/скелета!)·`Curve-follow`·
  **`Flow-advect`** (сэмпл motion/flow-карты видео как velocity-поля — «взмах сдувает тысячи точек»,
  паттерн №1 body-жанра). Noise-нода обязана уметь выдавать Gradient и Curl (урок TD).
- **Emitter** (нода): рождение из PointSet/продюсера (руки/лица/маска людей/`scatter-from-depth`)/формы;
  rate/burst; наследует ЦВЕТ видео и СКОРОСТЬ движения. + **target-позиции на силуэте с lerp к ним** —
  иконический эффект «тело рассыпалось/собралось».
- **Scatter** (нода): где рождать — rejection-sampling по карте плотности (`Depth`/маска/noise) + blue-noise дизер.
- **Collide** (нода): об карту ГЛУБИНЫ — **screen-space по рецепту Niagara/Unity VFX Graph** (проекция →
  сэмпл depth → сравнение Z с допуском по радиусу → отражение вокруг нормали из градиента глубины;
  bounce/stick/kill). Дёшево, бэкенд-агностично, low-accuracy — это осознанный компромисс, точные коллизии
  не делать.
- **Light частиц** (дёшево на масштабе): псевдо-нормаль из скорости → Lambert; fresnel/rim; **depth-fade**;
  soft-particles; **цвет/яркость по age/velocity через РАМПУ → emissive → bloom** (формула «не голых точек»,
  §3.6.11.3). Реальные shadow-map'ы на миллионы — нет.
- **Рендер:** `THREE.Points` (1 вершина/частица) + аддитив/premultiplied, `depthWrite=off`, **БЕЗ сортировки**
  (коммутативно; сортировка — только для alpha-дыма, не берём в базу). `InstancedMesh`-квады — отдельный
  режим для stretched-billboards/рибонов, позже. **Бюджет = филлрейт, не вершины:** частицы 1–4 px; крупные
  мягкие — рендер в даунсэмпленный таргет + композит (GPU Gems 3 ch.23).
- **Цифры (верифицированы, §3.6.11):** 1М@60fps на дискретном GPU — подтверждённая норма (продакшн Expo-2025
  1М на three.js WebGPU; 2М на WebGL-GPGPU; официальный пример three.js — 500k). iGPU — 100–300k.
  → **адаптивный счётчик:** старт 256–512k, поднимать по замеру кадра (`core/perf.ts`).
- **Частоты — §0.1 работает и тут:** compute-апдейт — 60fps (дёшев); карты-входы (depth/motion) обновляются
  на своей низкой частоте, частицы интегрируют по последней карте. Эмиттер-семплинг карт — целиком на GPU.
- **Маппинг на код:** `pointSources`-продюсеры = эмиттеры; `depth`/`motion`-карты = scatter/forces/collide;
  bloom = свечение. WebGPU уже в проде (глубина transformers.js) — бэкенд знаком.

непрерывная сессия). Прочие лимит-сейверы: меньше повторных чтений/верификаций, тугой scope задачи, не держать
лишнюю историю. (Под API это смягчал prompt-cache; на подписке расход за окно всё равно идёт.)

### 3.6.11. Разведка 2026-06-10 (7 агентов: Houdini/TD/Blender устройство+практика, web-частицы) — выводы

Верифицированные факты, на которых стоят ревизии §3.6.6/§3.6.9. Полные конспекты с URL — в сессии аудита
2026-06-09/10; ключевое продублировано в memory `node-research-2026-06`.

**1. Архитектурные уроки (все три пакета сходятся):**
- **Явные рёбра + pull-вычисление.** Houdini кукает только display-флаг и его зависимости; TD — pull + dirty +
  явная time-dependence. «Что считать» определяет ПОТРЕБИТЕЛЬ (достижимость от Экрана), а не включённость
  продюсеров. Наша авто-активация (`PRODUCER_ENABLE`) — полумера push-модели; её баги (фантомные детекторы,
  op-цепочки не будят продюсеров, нетранзитивные конвертеры) исчезают как класс при компиляторе от рёбер. → T0e.
- **Per-element поля — ядро парадигмы, не добавка.** История Blender: 2.92 «явные Attribute-ноды» (= наш
  текущий скалярный Field + SetAttr) признаны тупиком и выброшены за один релиз ради fields (поле = функция,
  вычисляемая per-element в контексте потребителя). Selection-вход, Group/Split, wrangle, силы частиц стоят
  на этой модели. Чем больше нод построим на скаляре — тем дороже миграция. → T1-3 поднят в приоритете.
- **Состояние — только внутри явной солвер/фидбэк-границы** (Houdini Solver SOP с входом prev_frame; Blender
  Simulation Zone: state течёт через пару Input→Output + dt + Skip + кеш на выходе = бесплатный freeze-frame;
  TD Feedback POP). Остальной граф — чистая функция. Разделять «итерацию во времени» (солвер) и «итерацию в
  кадре» (repeat-зона).
- **Видимость данных — фича №1, не полировка** (Houdini MMB-инфо+Geometry Spreadsheet; Blender Viewer+спредшит —
  главная жалоба GN-сообщества «поля невидимы»; TD per-node cook-time+Probe+превью картинки НА ноде). → трек T-Debug.
- **Прочее подтверждённое:** bypass = pass-through (не «выключить ветку»); Null-нода как именованный чекпойнт
  биндингов; стабильный `id` отдельно от индекса; Reduce/Broadcast (Attribute Promote) — явная нода; чистка
  отработавших атрибутов; enum нельзя запитать проводом — не раскармливать `OpKind`; Node Groups (подграф с
  интерфейсом сокетов) — несущая фича («своя нода без кода», на ней вся экосистема Blender-ассетов); wrangle
  («формула на точку», `@attr`) — конечная точка эволюции, компилируется и в CPU-цикл, и в GPU-кернел.
- **Порядок миграции подтверждён чужой кровью:** Blender провалил «сразу универсальный частичный солвер»
  (Everything Nodes → on hold навсегда) и выиграл путём «статический граф → поля → state-зоны». Не
  перепрыгивать к T5-нодам до per-element полей (спайк-прототип — можно и нужно, см. T5).

**2. Частицы web 2026 (основания §3.6.6):** TSL транспилируется в WGSL И GLSL (официально; `WebGPURenderer`
сам падает в WebGL2, `compute()` через transform feedback; БЕЗ атомиков/shared memory на фолбэке);
`WebGPURenderer` не поддерживает `ShaderMaterial`. Цифры: 1М@60 дискретный GPU — норма (Expo-2025 продакшн 1М;
2М WebGL-GPGPU; three.js пример 500k), iGPU 100–300k. Соседи (grid+sort) — WebGPU-only (boids: миллионы).
WebGPU-поддержка ~82% (Chrome/Edge давно, Safari 26 default, Firefox 141/145 частично) — фолбэк ещё нужен.

**3. Практика «красивого» (TD-туториалы, Entagma/Houdini-моушн, body-жанр):**
- **Формула „дорогого" визуала живёт в обвязке, не в ядре алгоритма:** сглаживание (Lag attack/release на
  каждом живом сигнале) + палитра (Lookup←Ramp по люме, форма всегда ч/б до колоризации, НЕ raw RGB) +
  пост-«плёнка» (bloom/CA/grain/vignette) + **вариация=атрибут** (`rand(id)→fit→ramp` на pscale/Cd —
  равномерность читается как «дёшево/CG») + память (feedback). **Рампа — главный творческий контрол вообще**
  (ширина лент, цвет по age/velocity, falloff'ы) → рампа на КАЖДЫЙ маппинг атрибута, не только на драйверы.
- **Один сигнал → много параметров с разными remap-кривыми** = «живое» (Akten: скорость+ускорение+кривизна
  жеста → букет свойств отклика).
- **Рецепт №1 экосистемы — Feedback-петля** (см. §3.6.3 ⭐⭐). Самый дешёвый способ «органики»: система с
  памятью.
- **Body-жанр (наша ниша; Krueger: «Response is the medium»):** тело = непрерывное поле сигналов; отклик —
  через автономную систему с инерцией (пружина/затухание/частицы с массой), НЕ 1:1-копирование. Канон жанра
  и наша готовность: plexus=constellation✓ · тепловизор✓ · point cloud+скан по Z✓ · Posy motion-extraction
  (≈наша разница кадров) · частицы от optical flow (нужен НАПРАВЛЕННЫЙ flow — сейчас только энергия) ·
  силуэт→эмиттер со сборкой к target · маска-стенсил «мир внутри силуэта» · светящийся контур+аура ·
  slit-scan по глубине. Сглаживание точек — One Euro (есть ✓).
- **Пользователь мыслит референсом/жанром, не нодами** → пресеты-рецепты (готовый подграф с 2–3 рампами
  наружу) закрывают 80% желаний; механизм — Node Groups (после GraphDoc), до них — пресеты текущего формата.
- **Дефолты = эстетика:** smooth на живых сигналах по умолчанию; лёгкий rand(id)-разброс размера новых точек;
  цвет частиц по age/velocity→emissive→bloom; depth-fade; кнопка randomize-seed на каждой шумовой ноде
  («happy accidents» — названный метод работы художников).

### 3.6.12. Particle-ноды v2 — СПЕК СИСТЕМЫ (ревизия §3.6.6; ресёрч + решения пользователя 2026-06-10)

> **Установка пользователя (главная):** нужны НЕ фичи («частицы из глубины»), а **композиционная
> нод-система** уровня Houdini/TD. Emitter рождает частицы НА ТОМ, что в него воткнули (точки,
> поверхность, карта); силы, цвет, группы, спрайты — ноды. Примеры-валидаторы: «глубина×нойз→
> выдавливание→эмит», «YOLO-вырезка→перекраска→эмит с вырезанного», «куб←руки→нойз-скаттер→эмит→
> антигравитация→цвет ртом→split→спрайты на часть» — ВСЕ обязаны собираться цепочками, не кодом.
> **И всё это — РЕАЛТАЙМ** (60fps живого видео, не оффлайн-кук). Спайк T5 (§3.6.9) — фундамент.

**Ресёрч-основание (2026-06-10, 2 агента; Houdini 21.0 локальные доки + docs.derivative.ca TD POPs).**
Конвергенция обеих систем:
1. **Один владелец симуляции** (Houdini POP Solver / TD Particle POP): буферы, интеграция, age/смерть.
2. **Силы аккумулируют force-атрибут СЛОЖЕНИЕМ** (Houdini `force`, TD `PartForce` «added to input's») →
   коммутативны, порядок не важен, набор расширяем без правки солвера.
3. **Эмиттер рождает с входной геометрии** (Houdini emission types All Points/Points/Scatter onto
   Surfaces + Emission Attribute-модуляция; TD «born at input point positions», Birth Rate | Birth Attribute).
4. **Смерть = флаг** (TD `PartDeath`), население капнуто — ровно age-recycle модель нашего спайка.
5. **Цвет/группы = атрибут-ноды** (POP Color constant/random/ramp-by-age/blend; POP Group rule/bound/random).
6. **Спрайты/инстансы — ПОСЛЕ сима** (POP Instance/Sprite; TD instancing из POP-атрибутов).
7. **Данные: SoA-буфер на атрибут, GPU-resident** (TD; зарезервированные PartId/P/PartVel/PartAge/
   PartLifeSpan/PartForce/PartMass/PartDeath) — модель нашего спайка (`instancedArray`).

**Сокет-тип `particles` (новый, 4-й; оранжевый).** GPU-домен: стейт в VRAM, в CPU-ноды НЕ течёт
(как сим-данные Houdini/TD — улица с односторонним движением CPU→GPU). Цепочка:
`Emitter → [Force…] → [Цвет/Группы…] → Render(точки|спрайты)`. Force-ноды декларативны
(cook-time ничего не считают — система собирает набор и компилирует TSL). Рендер — WebGPU-канвас
(вывод спайка: WebGL2-фолбэк не рисует из storage-буфера).

**МНОЖЕСТВО ЭМИТТЕРОВ — ОБА РЕЖИМА (решение пользователя 2026-06-10):**
- **Режим 1 — независимые системы (v1, Phase A; основной кейс «несколько эмиттеров»):** каждый
  **Emitter = самодостаточная система** (TD-модель Particle POP: рождение+смерть+age+интеграция
  фьюзятся в эмиттер — owns свой буфер/maxParticles/damping). У КАЖДОЙ свои Force/Color/Render.
  Бросил ещё один Emitter → ещё одна независимая система (пример пользователя: глубина+curl ОДНА
  система ∥ руки+антиграв+спрайты ДРУГАЯ — одновременно, разные силы/рендер). **Все системы рендерятся
  в ОДИН общий WebGPU-канвас** (один `renderer.render`, N объектов `Points` в одной сцене — дёшево).
- **Режим 2 — слияние в одну популяцию (Phase D, Houdini popnet):** **Merge-particles**-нода
  объединяет несколько Emitter-выходов в ОДНУ популяцию ДО сил → общие Force на всех, разные эффекты
  по частицам — через **Group/Split частиц** (силы/цвет, скоупленные группой). Нужен Group/Split
  частиц (Phase D) — поэтому merge там.
- **Модель данных forward-compat под оба:** `ResolvedParticleSystem.emitters[]` — МАССИВ источников
  рождения (v1 = ровно 1 на систему; Merge в Phase D кладёт несколько в один). Так Phase D не
  переписывает модель. Solver-параметры (maxParticles/damping/timeScale) живут НА системе
  (в v1 = на эмиттере; при merge — на Merge-узле-владельце популяции).
- **Бюджет реалтайма делится:** адаптивное население (EMA-счётчик) распределяет общий бюджет частиц
  по активным системам — 4 эмиттера по 64k, не 4×256k. 60fps приоритетнее суммы.

**Ноды Phase A (params из ресёрча, v1-минимум):**
- **Emitter** — рождает НА ТОМ, что воткнуто. **ДВА входа** (решение пользователя):
  `points` (PointSet: hands/scatter/split(маска)/transform/displaced-грид Phase C/любой продюсер) →
  эмит с точек; `map` (Map2D: глубина/нойз/маска/комбинации Phase B) → эмит ПО КАРТЕ-ПЛОТНОСТИ —
  **rejection-sampling прямо на GPU при респавне** (карта уже DataTexture, без CPU-скаттера;
  «плейн с нойзом → эмитить по нойзу» = чисто GPU); **оба** → эмит с точек, карта модулирует
  вероятность (Houdini Emission Attribute); ничего → box-фолбэк (спайк). Params: `rate` (частиц/сек) ·
  `life`+`lifeVar`(доля) · `jitterPos` · `initSpeed`+`velSpread`. **Наследование Cd источника**
  (SetAttr(Cd)→Emitter = цветные при рождении; map-вход — цвет сэмпла). Наследование v — defer.
- **Force — ОДНА нода с секциями** (решение пользователя; прецедент TD Force Radial POP: radial/axial/
  spiral/planar+gravity+wind в одной ноде с enable на блок; = принцип «секции/enum — вариации одного
  действия „приложить силу“»). Секции, каждая `enable`+params: **Гравитация** (вектор gx,gy,gz —
  антигравитация = вверх; strength) · **Шум** (режим `curl`|`turbulence`; amplitude · swirlSize ·
  animSpeed; roughness/octaves v2; **честный флюид = сеточный солвер, отдельный трек; curl — его
  дешёвая реалтайм-форма**) · **Drag** (airResist, targetv=0). Несколько Force-нод чейнятся
  (аккумуляция, коммутативно). **Attract** (цели=точки рук, POP Attract) — ОТДЕЛЬНАЯ нода в Phase D
  (свой points-вход целей, другая сигнатура).
- **Solver-роль** — `maxParticles` · `timeScale` · `damping`. Владеет буферами, semi-implicit Euler
  (спайк), age-recycle. Rate-обузданный респавн: мёртвая рождается с вероятностью ≈ rate·dt/deadCount
  (element-wise приближение, без атомиков/компакции). **v1 (Режим 1): фьюзится в Emitter** (TD-модель —
  отдельной ноды Solver нет, эмиттер = система). **Режим 2 (Phase D): standalone владелец популяции**
  (Merge-particles-узел), куда сливаются несколько эмиттеров.
- **ParticleColor** — режимы `byAge` (ramp 2 цвета) / `byVelocity` (спайк) / `constant`. Рендер-стадия
  (colorNode), не compute. Field-вход — v2.
- **Render** — `size` · `opacity` · `blend` (additive/normal). Терминал домена.
- **Field-вход на amplitude/strength сил** (модуляция: рот/аудио/энергия движения) — скаляр-на-кадр
  через существующий `resolveSignal` → uniform. Per-element поле НА ЧАСТИЦУ = перенос Field-алгебры
  в TSL — отдельный трек, честно деферим.

**Мост CPU↔GPU (ключевая архитектура):**
- **Спавн-буфер:** emitter-вход points = PointField (CPU, тот же `fieldForLayerId`; у ParticleCanvas
  появляются refs как у ModifierOverlay) → Float32Array [x,y,z,r,g,b]×N → storage-буфер; обновление
  на низкой частоте/по изменению (паттерн DataTexture, PointCloud.tsx). GPU-респавн сэмплит случайный
  индекс + jitterPos.
- **Координаты и камера (решение пользователя):** частицы в video-aligned пространстве (0..1
  contain-фит, как все оверлеи). Камера **как у облака**: дефолт фронтальная орто на кадр
  (frontLock, CloudCamera-паттерн) **+ переключение в свободную орбиту** (OrbitControls). Деталь:
  канвас частиц сейчас pointerEvents:none — при unlock включать ввод (или шарить орбиту с cloud).
- **Сборка систем:** `resolveConfig` эмитит `particleSystems: ResolvedParticleSystem[]` {id,
  **emitters: [{params, sourceRef, mapRef}]** (массив — v1 ровно 1, Merge кладёт несколько),
  solver{maxParticles,timeScale,damping}, forces[{kind,params}], color, render} — обход
  particles-цепочки по рёбрам (прецедент `computeFxOrder`/`fxChain`). Particle-модуль держит МАССИВ
  сим-инстансов (по одному на систему), диффит СТРУКТУРНУЮ сигнатуру каждой → пересборка TSL (редко,
  асинхронно); params → uniforms (каждый кадр). Все системы — в одном канвасе/сцене (N `Points`).
- **Авто-активация:** Emitter форс-поднимает source-продюсер (существующий `PRODUCER_ENABLE`).

**РЕАЛТАЙМ-ИНВАРИАНТЫ (сквозное требование пользователя; нарушение = блокер ревью):**
1. **Сим+рендер — каждый кадр, дёшево.** Спайк: 200k+ частиц + curl(6 тапов) + рендер = 16.7ms WebGPU.
   Сила = element-wise добавка; бюджет v1 ~4 силы на 256k @ 60. **Адаптивное население** (EMA-счётчик
   спайка) наследуется: при просадке кадра население ужимается — 60fps приоритетнее числа частиц.
2. **Крутка параметров НИКОГДА не пересобирает шейдер.** Все params = uniforms → мгновенный отклик.
   Пересборка TSL — только при смене СТРУКТУРЫ (добавил/убрал Force-ноду), редкая и **асинхронная**
   (`renderer.compileAsync`): старый сим крутится, пока собирается новый, без фриза.
3. **Разделение частот (§0.1).** CPU-граф (детекторы, fieldForLayerId, спавн-буфер) — на своей частоте
   (15–30Гц); GPU-сим интегрирует на 60 по ПОСЛЕДНЕМУ буферу. Сим не ждёт CPU. Буфер мал (десятки КБ).
4. **Ноль GPU→CPU readback в кадровом цикле.** Превью/дебаг частиц — только из рендера.
5. **Никакого нового per-frame обхода графа.** Emitter-источник переиспользует throttled-резолв
   точек оверлеев/statsRef — не вторая прокачка.
6. **Map2D-кит (Phase B) под реалтайм:** для шейдерного потребления — GPU-проходы (FBO-цепочка есть);
   для CPU-потребителей (scatter) — низкая частота как глубина сейчас. Превью на нодах — мини-канвас
   ≤96px, throttled ~5Гц (паттерн PointMeters), НЕ в hot-path.

### 3.6.13. Направление U — УНИФИКАЦИЯ точек и частиц (установка пользователя 2026-06-11)

**Установка:** «точки и частицы — принципиально одно и то же; форсы должны накидываться и на облако».
Это Houdini-модель (POP-солвер двигает обычные точки геометрии; в hhh.hip popnet берёт points и отдаёт
points). **Концепт:** частица = точка с атрибутами (`P, v, age, id, Cd, pscale`). Различие НЕ в типе,
а в РЕЗИДЕНТНОСТИ буфера: CPU-PointSet (сотни–тысячи, полная поэлементная свобода: Split/Sort/SetAttr/
спредшит/Линии) vs GPU-популяция (сотни тысяч, состояние в storage-буферах, ноль readback — TD-модель).
Сокеты `points`/`particles` — это сегодняшняя честная маркировка ГРАНИЦЫ СТОИМОСТИ (GPU→CPU readback
рвёт 60fps), а не разных сущностей. Цель U — чтобы граница ощущалась как резидентность ОДНОГО типа,
с одинаковым словарём операций по обе стороны.

**Фазы:**
- **U0 ✅ (2026-06-11): «Силы (точки)» (`pointForce`)** — CPU-солвер сил над ЛЮБЫМ PointSet, ВТОРОЙ
  stateful-конвертер по шаблону Trail (модульная карта `pfState` {p,v,age} по стабильному `id` +
  frameTick-барьер; снапшот-потребители не мутируют). Секции В ПАРИТЕТЕ с GPU-Force — переиспользует
  ТЕ ЖЕ `force*`-поля params (гравитация / curl|turbulence по 2D-fbm конечными разностями / drag /
  границы bounce|wrap в [0,1]²) + `pfStrength` (масштаб шага, ср. math1.gain TD) + `pfLife` (респавн
  на исходную точку, аналог over1/multiply1; 0=∞). Теперь «форсы на облако»: `Видео.глубина → Scatter →
  Силы(точки) → Линии/Splat/Emitter` — облако глубины, гонимое теми же силами, что GPU-частицы.
- **U1 ✅ (2026-06-11): 3D-точки сквозь CPU-тракт.** Scatter пишет `z = (глубина 0..1)·scatterZScale`
  во ВСЕ точки при флаге `scatterZ` (даже z=0 на дальних — потребитель единообразно в 3D-ветке);
  рисовалки уважают z: `drawLines.map(nx,ny,z?)` — при `z===undefined` плоский contain-фит байт-в-байт
  (2D-продюсеры не тронуты), при z задан — проекция через камеру облака (`App.cameraRef`, мир как
  Constellation3D: `x=(1-2nx)·aspect, y=1-2ny, z=-(z·depthScale)`), а без активного облака —
  `fakePerspective`-параллакс (спред от центра по `z·depthScale`). pointForce уже интегрирует z
  (сохраняет/респавнит). z-конвенция когерентна: и CPU-проекция, и GPU Emitter-мост (`uvToWorld·
  uDepthScale`) дают магнитуду `depth·scatterZScale·depthScale` — по одному scatterZScale и depthScale
  на ветке. **Гоча (фикс):** `CloudCamera` теперь зануляет `cameraRef` в cleanup — иначе always-on
  ModifierOverlay проецировал бы 3D через ЗАМОРОЖЕННУЮ cloud-камеру после cloud→flat.
- **U2: один сокет/словарь.** `particles`-сокет → «points (GPU)» — оттенок того же типа; Emitter =
  «переезд CPU→GPU» (уже работает: spawn-текстура).
  - **U2-b ✅ (2026-06-11): ЕДИНАЯ нода «Force»** — авто-диспатч по РЕЗИДЕНТНОСТИ входа. `force`-вид
    стал ДВОЙНЫМ доменом: `isParticle("force")` И `isPointProducer("force")` одновременно. Воткнут
    `particles` (Emitter→Force→Render) → GPU-сила (buildParticleSystems, как было); воткнут `points`
    (pointForceSource, convertInput optional) → CPU-солвер (fieldForLayerId `case "force"` = ТОТ ЖЕ
    `pointForceToField`, что U0). force*-поля общие; pfStrength/pfLife — CPU-ветка (GPU игнорит). domain
    остаётся "particle" → nonVisual (НЕ флипает any2D — GPU-режим не плодит композит; CPU-режим
    покрыт visual-апстримом grid/scatter); applyGraph particle-ветка читает convertInputs (синк
    pointForceSource из points-ребра); enabled НЕ гейтится отсутствием points (GPU жив без него).
    GraphView: force строится ТОЛЬКО в particle-проходе (+points-сокеты) — producer-проход пропускает
    domain particle (иначе дубль-нода). App: CPU-Force НЕ будит `particlesEnabled` (иначе lone-Force →
    box-демо). `pointForce`-вид сохранён резолвящимся (back-compat U0/U1), убран из палитры. **Инвариант
    U соблюдён: НЕ слияние бэкендов — один словарь/UX, два проверенных пути по резидентности.** Дальше
    U3: атрибуты Cd/pscale/v сквозь границу.
  - **U2-a ✅ (2026-06-11): конвертер «Particles→Points»** = явный переезд GPU→CPU. Нода
    `particlesToPoints` (domain "convert", point-producer): вход — `particles`-сокет от Render-ноды
    (socket-crossing; Render получил выход-«отвод» рядом с терминалом), выход — обычный PointSet.
    Движок: `ParticleSystem.readback(renderer,maxPoints)` — `getArrayBufferAsync(positionBuffer)`
    (vec3 stride=4 floats, padding до 16б; проверено live на WebGPU), downsample по живому count,
    world→норм-UV (инверсия uvToWorld: nx=(wx/rectX+1)/2, ny=(1−wy/rectY)/2, d=wz/depthScale), kill
    мёртвых (z<−100). Мост `ParticleFieldCanvas` на ~7Гц (busy-guard, ноль readback в кадре рендера)
    читает системы, на которые ссылаются конвертеры (`renderId→sysId` через `ResolvedParticleSystem.
    renderId`), и кладёт PointField в МОДУЛЬНЫЙ `particleReadbackStore` (pointSources); резолвер
    `particlesToPoints` читает стор по `particlesSource`(=id Render). disposed/alive-гарды вокруг
    await, прун стора на тик и анмаунт. **socket-обобщение convertInputs:** `ci.socket??"points"` —
    единственная правка graphDoc (synthesize/edgeForLayerParam; applyGraph уже socket-agnostic). Теперь
    над GPU-частицами весь CPU-кит: Lines/Split/Sort/Trail/спредшит; z несётся (U1-проекция). СТОИМОСТЬ:
    readback тянет ВЕСЬ буфер (cap×16б, ~4.8МБ@300k) — бюджет режет ВЫХОД, не трансфер; ~7Гц «как кадр
    глубины». Дальше U2: склейка GPU-Force+pointForce в одну ноду «Force».
- **U3: общие атрибуты сквозь границу.** Cd/pscale/v едут в обе стороны.
  - **U3-a ✅ (2026-06-11): Cd GPU→CPU.** Particles→Points readback тянет ВТОРОЙ буфер (`colorBuffer`)
    по флагу `p2pColor` — выход страйд-6 [nx,ny,d, r,g,b]×k вместо страйд-3; `readbackToField(flat,
    withColor)` строит векторный атрибут `Cd` (comps:3) → `drawLines` красит Lines/constellation цветом
    частицы. Замыкает круг «цвет видео → частица (Cd CPU→GPU) → readback (Cd GPU→CPU) → цветные линии».
    Флаг по умолчанию off (второй getArrayBufferAsync = ×2 трансфер; opt-in). lockstep: withColor у
    движка и readbackToField ведётся одним `c.params.p2pColor` в мосте; нет colorBuffer → пусто (не
    рассинхрон страйда). disposed-гард после обоих await.
  - **Отложено:** pscale GPU→CPU невозможно без per-particle size-буфера (размер сейчас uniform uSize);
    v GPU→CPU (velocityBuffer есть) — нет CPU-потребителя, отложено; pscale/v CPU→GPU (spawn-формат
    pos+col → +size/vel) — Phase D. group/split GPU-популяции (Phase D) — та же модель критерия-поля, что CPU-Split.

**Инвариант U:** унификация — это словарь и UX, НЕ слияние бэкендов: CPU-тракт никогда не получает
GPU-буфер «как массив» (только явный конвертер с бюджетом), GPU-тракт не зовёт JS per-element.

### 4.2. PointSet (SOP/POP-подобные)
`Transform` · `Merge` · `Filter/Group` (по атрибуту: `conf>0.5`, `group==face`) · `Jitter/Noise` ·
`Connect/Lines` (топология → рёбра: native/chain/nearest) [есть, обобщить из эффекта-линий] ·
`Copy-to-Points` (спаун меша на каждой точке) · `Trail` (история N кадров) [нов]

### 4.3. Field (CHOP-подобные — основа в `drivers`)
`Reduce` (PointSet/Map → скаляр: count, центроид, bbox, средняя люма) [есть, `drivers.ts`:
`motionEnergy` (площадь-взвешенная активность боксов), `faceCentroid` (faceX/faceY — центр боксов
лиц), `peopleCountNorm` (count людей / PEOPLE_MAX)] · `Lag/Filter` (есть: `oneEuro`) · `Math/Remap`
(в биндингах: `lo/hi` диапазон + `curve` нелинейный отклик ease-in/out, `drivers.shapeT`;
**standalone-нода Math на проводе [есть]** — `OpNode`+`resolveSignal`, цепочка Field-нод, см. §3.5.5 C-Math) · `Trigger` (порог → импульс/ADSR) [нов]

### 4.4. Конвертеры — главный клей
- **`Splat/Rasterize`**: PointSet → Map2D (форма на каждой точке) **[есть]** — нода `splat`
  (`modifierRegistry`): `SplatMask` (`ui/SplatMask.tsx`) растеризует форму (`renderSplat.drawSplats`)
  в offscreen-Canvas2D → `THREE.CanvasTexture` → видео-шейдер (`flatShader composeSplat`, uSplat).
  Режимы: overlay-цвет / alpha-вырез видео. Текстура шарится через `SplatApi`-реф (App→Scene→
  FlatView). §0.2: один аплоад/кадр, малый N. Композит-режим — по первому splat-эффекту.
- **`Scatter`**: Map2D → PointSet **[есть]** — `pointSources.scatterToField` (прорежённая сетка по
  глубине/маске + порог; нода «Scatter», вход `глубина`, см. §3.5.5 C-Scatter). **`Sample`**: PointSet×Map2D
  → PointSet **[есть]** — `pointSources.sampleToField` (сэмплит глубину в `value` точек продюсера; нода
  «Sample», входы `точки`+`глубина`, см. §3.5.5 C-Sample)
- **`Reduce`**: PointSet|Map2D → Field **[есть]** — `drivers.ts` (motionEnergy/faceCentroid/peopleCount), §4.3
- **`Render`**: 3D-PointSet → Map2D (рендер three.js-сцены в текстуру, композит назад в 2D) [инфра есть]

Пример («белые квадраты на движении как альфа видео») **— реализован**:
`motion.points ─Splat(square, mode=alpha)→ маска ─(uSplat: col*=s.a)→ видео сквозь формы`.
Полный путь `PointSet→Map2D→видео-композит` рабочий; следующий шаг — Splat→Mask(альфа=люма) как
отдельный оператор-нода (сейчас composite-режим зашит в шейдер, не отдельная Mask-нода).

## 5. Кит форм (параметрические, не картинки)

Форма — дескриптор `{kind, params}`. Один дескриптор питает оба оператора: в 2D — SDF-спрайт
для `Splat`, в 3D — параметрический меш для `Copy-to-Points`. Per-point атрибуты (`size`, `Cd`,
`vel`, `age`) разводятся в per-instance параметры → вариативность даром.

### 5.1. 2D — SDF во фрагментном шейдере (масштаб-независимо, AA через `fwidth`)

| Форма | Параметры |
|---|---|
| `disc` (полигон по числу сторон) | `sides 3..∞`, `radius`, `round` |
| `box` | `w`, `h`, `corner` |
| `star` | `points`, `rIn`, `rOut` |
| `superformula` | `m`, `n1,n2,n3` |

Универсальные модификаторы: `stroke(width)` → `d = abs(d) - width` (кольцо/контур любой
формы); `fill`↔`outline`; `rotation`; `softness`.

```glsl
float sdPoly(vec2 p, float r, float n){            // правильный полигон по числу сторон (IQ)
  float an=3.14159265/n, he=r*tan(an);
  p=-p.yx; float bn=an*floor((atan(p.y,p.x)+an)/an);
  vec2 cs=vec2(cos(bn),sin(bn)); p=mat2(cs.x,-cs.y,cs.y,cs.x)*p;
  return length(p-vec2(r,clamp(p.y,-he,he)))*sign(p.x-r); }
float sdBox(vec2 p, vec2 b, float r){ vec2 d=abs(p)-b+r;
  return min(max(d.x,d.y),0.)+length(max(d,0.))-r; }
float onion(float d, float w){ return abs(d)-w; }  // дырка/контур
```

### 5.2. 3D — параметрические примитивы (instanced)
- Меши `box/sphere(икосаэдр)/cylinder/cone/torus/plane` через `InstancedMesh` (один draw-call
  на тысячи инстансов).
- **Суперэллипсоид** (одно уравнение → много форм): `(|x|^(2/e)+|y|^(2/e))^(e/n)+|z|^(2/n)=1`
  (`e=n=1` сфера; `e=n→0` куб; `e=1,n→0` цилиндр). Генерится раз на CPU, инстансится.

### 5.3. Текст
- Быстрый путь: bitmap-атлас (`buildAtlas` в `ScanAscii3D`/`FlatView`) [есть].
- Качество: **MSDF** (чёткий на любом масштабе/угле) — ещё один `kind` формы (Splat в маску /
  спаун на точки).

## 6. Realtime-3D (вращаемое облако, тени, спаун на точки)

- **Одна three.js-сцена для всех PointSet** с общей камерой → общий depth, трансформы, тени,
  единый orbit. (Сейчас 3D-оверлеи лепят `innerHTML` каждый кадр — это P1.)
- **2D-продюсеров поднимаем в 3D через глубину:** сэмплируем карту глубины в UV лица/рук/боксов
  → у них появляется Z → живут в одном мире с облаком.
- **Спаун:** `InstancedMesh` из `PointSet.P`. Быстрый путь — WebGPU compute + TSL (P5), запасной
  — WebGL `InstancedMesh`.
- **Тени дёшево:** directional light + PCF shadow map пониженного разрешения только на инстансах
  + контактная тень на плоскости.
